# SSH Config Master

## 产品概述

SSH Config Master 是一款纯粹的 SSH 配置管理工具，专注于帮助开发者高效组织、编辑和同步 SSH 连接配置。它不内置终端模拟器或连接功能，而是通过生成标准的 `~/.ssh/config` 文件，让用户自由选择任意终端工具（iTerm2、Windows Terminal、VSCode Remote SSH 等）进行连接。通过直观的图形界面和强大的命令行支持，解决多服务器配置管理的混乱问题。

### 双模式界面

#### GUI 图形界面

- **分组树形视图**：拖拽即可重组服务器分组
- **配置编辑器**：表单化编辑 SSH 配置，支持所有原生选项
- **实时预览**：编辑时实时预览生成的 `~/.ssh/config` 内容
- **快速启动**：点击主机卡片调用系统终端或复制连接命令
- **批量操作**：支持多选进行批量导入、导出、删除、分组
- **主题支持**：支持浅色/深色/系统跟随模式

#### CLI 命令行界面

```bash
# 基本命令
sshm list                      # 列出所有配置
sshm add <name> --host <ip>     # 添加新配置
sshm edit <name>                # 编辑配置
sshm delete <name>             # 删除配置
sshm show <name>                # 查看配置详情
sshm copy <name>                # 复制 ssh 命令到剪贴板
sshm open <name>                # 调用系统终端连接
sshm config <name>              # 输出 ssh config 格式

# 分组管理
sshm group create <group>      # 创建分组
sshm group add <group> <name>   # 添加到分组
sshm group list                 # 列出所有分组

# 同步
sshm sync                       # 同步到 ~/.ssh/config
sshm sync --watch               # 监听变更自动同步

# 导入导出
sshm import <file>              # 导入配置
sshm export [--format json|yaml|ssh] # 导出配置
```

### 快速启动机制

本产品**不内置终端连接功能**，而是通过以下方式桥接到用户习惯的终端工具：

| 启动方式 | 行为 | 适用场景 |
|---------|------|---------|
| **调用系统终端** | 自动执行 `ssh <name>`，在系统默认终端中打开 | GUI 一键启动 |
| **复制连接命令** | 将 `ssh -i ~/.ssh/key user@host -p 22` 复制到剪贴板 | 粘贴到任意终端 |
| **同步到 ~/.ssh/config** | 写入标准 SSH 配置文件，用户可在任何终端用 `ssh <name>` 连接 | 通用方式 |
| **VSCode Remote SSH** | 自动同步分组配置到 VSCode 识别的 config 文件 | VSCode 远程开发 |

**设计哲学**：做配置的「单一数据源 (Single Source of Truth)」，让连接这件事回归到用户最顺手的终端工具。

## 技术架构

### 跨平台技术选型依据

#### GUI 框架选型：Tauri 2.0 vs Electron vs Qt vs Flutter

| 评估维度 | Tauri 2.0 | Electron | Qt (PyQt/Qt) | Flutter |
|---------|-----------|----------|---------------|---------|
| **安装包体积** | ~3-8 MB | ~80-150 MB | ~30-60 MB | ~15-30 MB |
| **运行时内存** | ~20-50 MB | ~150-300 MB | ~30-80 MB | ~50-100 MB |
| **平台支持** | Win/Mac/Linux/Mobile | Win/Mac/Linux | Win/Mac/Linux | Win/Mac/Linux/Mobile/Web |
| **原生 API 访问** | ✅ Rust FFI 零开销 | ❌ 需 Node 桥接 | ✅ 原生绑定 | ⚠️ 需插件通道 |
| **安全模型** | ✅ 最小权限、IPC 白名单 | ❌ 完整 Node 运行时 | ✅ 系统原生 | ⚠️ Dart 隔离 |
| **CLI 共享核心** | ✅ Rust 共享同一份代码 | ❌ 需额外打包 | ⚠️ 需独立实现 | ❌ 需独立实现 |
| **前端生态** | ✅ 任意 Web 框架 | ✅ 任意 Web 框架 | ❌ QML/自定义 | ⚠️ Dart 生态 |
| **自动更新** | ✅ 内置 Updater | ✅ electron-updater | ⚠️ 需自实现 | ⚠️ 需自实现 |
| **系统托盘** | ✅ 原生支持 | ✅ 支持 | ✅ 原生支持 | ⚠️ 插件支持 |
| **开源协议** | MIT/Apache 2.0 | MIT | LGPL/商业 | BSD |

**选择 Tauri 2.0 的核心理由：**

1. **CLI + GUI 代码零成本共享**：本产品需要同时提供 GUI 和 CLI 两种交互模式。Tauri 的 Rust 后端可直接编译为独立 CLI 二进制文件，无需重写核心逻辑。Electron 和 Flutter 均无法实现这一点——Electron 的 CLI 分发体积过大，Flutter 的 CLI 需完全重写。

2. **安全敏感场景的天然适配**：SSH 配置管理涉及密钥和密码存储。Tauri 的安全模型基于最小权限原则，IPC 通信通过白名单严格管控，不存在 Electron 中完整的 Node.js 运行时攻击面。这对于一个管理 SSH 凭据的工具至关重要。

3. **开发者工具的体积敏感**：目标用户是开发者，他们对工具的"侵入感"高度敏感。3-8 MB 的安装包 vs 150 MB 的 Electron，差距达到 20 倍。一个 SSH 配置管理工具不应该比它管理的服务器还"重"。

4. **原生系统集成**：SSH 是系统级工具，需要读取 `~/.ssh/config`、管理系统托盘、注册 URI Scheme 等。Tauri 通过 Rust FFI 直接调用系统 API，无需经过 Node.js 桥接层，更可靠也更高效。

5. **Tauri 2.0 的移动端前瞻**：v2.0 已支持 iOS/Android，为未来可能的移动端预览功能预留了升级路径。

#### 前端选型：React + TypeScript vs Vue vs Svelte

| 评估维度 | React + TypeScript | Vue 3 + TypeScript | Svelte |
|---------|-------------------|-------------------|--------|
| 生态成熟度 | ✅ 最成熟 | ✅ 成熟 | ⚠️ 成长中 |
| 类型安全 | ✅ 完整 TS 支持 | ✅ 完整 TS 支持 | ⚠️ TS 支持改善中 |
| 组件库选择 | ✅ 极丰富 | ✅ 丰富 | ⚠️ 较少 |
| 开发者基数 | ✅ 最大 | ✅ 大 | ⚠️ 较小 |
| 包体积 | ⚠️ 较大 | ⚠️ 中等 | ✅ 编译时优化 |
| 学习曲线 | ⚠️ 中等 | ✅ 较平缓 | ✅ 最平缓 |

**选择 React + TypeScript 的理由：**

- 面向开发者的工具，React 生态和开发者基数最大，便于社区贡献
- TypeScript 提供完整的类型安全保障，对 SSH 配置这种结构化数据尤为重要
- Radix UI + TailwindCSS 可构建轻量且高可定制的组件，避免引入重型 UI 框架

#### 数据存储选型：SQLite vs 文件系统 vs IndexedDB

| 评估维度 | SQLite | 文件系统 (JSON/YAML) | IndexedDB |
|---------|--------|---------------------|-----------|
| 查询能力 | ✅ 完整 SQL | ❌ 需全量加载 | ⚠️ 索引查询 |
| 事务支持 | ✅ ACID | ❌ 无 | ⚠️ 有限 |
| 并发安全 | ✅ WAL 模式 | ❌ 需加锁 | ⚠️ 浏览器限制 |
| 数据量 | ✅ TB 级 | ⚠️ 百条级 | ⚠️ 受限 |
| 跨平台 | ✅ 单文件，通用 | ✅ 通用 | ❌ 仅浏览器 |
| 可移植性 | ✅ 单 .db 文件 | ✅ 纯文本 | ❌ 浏览器绑定 |

**选择 SQLite 的理由：**

- 单文件部署，便于备份和迁移
- WAL 模式支持并发读写，GUI 和 CLI 可同时操作
- 完整 SQL 查询能力，支持分组、标签等多维度检索
- Rust 生态有成熟的 rusqlite 绑定，性能优秀

#### CLI 框架选型：Clap vs StructOpt

**选择 Clap（v4）的理由：**

- Clap v4 已合并 StructOpt 的派生宏功能，无需二选一
- Rust 生态事实标准，文档完善，社区活跃
- 支持子命令嵌套，完美匹配 `sshm group add` 这类多级命令
- 自动生成 Shell 补全脚本（Bash/Zsh/Fish/PowerShell）

#### 加密方案选型：AES-256-GCM + Argon2

```
密码加密流程:

用户输入 Master Password
        │
        ▼
  Argon2id 密钥派生
  (内存: 64MB, 迭代: 3, 并行: 4)
        │
        ▼
  256-bit 加密密钥 (KEK)
        │
        ▼
  AES-256-GCM 加密数据密钥 (DEK)
        │
        ▼
  AES-256-GCM 加密实际数据
```

**选择理由：**

- **Argon2**：2015 年密码哈希竞赛冠军，抗 GPU/ASIC 暴力破解，替代老旧的 PBKDF2/bcrypt
- **AES-256-GCM**：认证加密，同时保证机密性和完整性，NIST 推荐标准
- **双层密钥架构**：更换 Master Password 无需重新加密所有数据，只需重加密 DEK

## 使用场景

### 场景 1：管理大量开发服务器

> 开发者小张有 20+ 台开发/测试/生产服务器，通过 SSH Config Master 分组管理，配置自动同步到 `~/.ssh/config`，在任何终端中直接 `ssh web-prod-01` 即可连接。

> 运维团队将 SSH 配置导出为加密文件，通过内部工具分发给新成员，新成员导入后立即可用。

### 场景 3：VSCode 远程开发

> 前端工程师打开工作区 "project-frontend"，SSH Config Master 自动将相关分组的服务器配置同步到 `~/.ssh/config`，VSCode Remote SSH 直接识别并可用。

### 场景 4：配置版本控制

> 将 SSH 配置导出为 YAML，通过 Git 管理，每次变更都有审计记录。新人入职 clone 仓库后一条命令导入。

## FAQ

**Q: 会收集用户数据吗？**
A: 不会。我们采用零信任设计，不收集任何遥测数据，所有数据存储在本地。

**Q: 可以直接连接服务器吗？**
A: 不内置终端。点击「启动」会调用系统默认终端执行 ssh 命令，也可以复制连接命令粘贴到任意终端。核心设计是做配置的单一数据源，让连接回归你最顺手的终端工具。

**Q: 支持 SSH Jump Host 跳板机吗？**
A: 支持，完整支持 ProxyJump 和 ProxyCommand 配置，同步到 ~/.ssh/config 后即可使用。

**Q: 如何保证密码安全？**
A: 使用 Master Password 加密存储，采用 Argon2 密钥派生和 AES-256-GCM 加密。

**Q: 可以同步到多台设备吗？**
A: v3.0 将支持端到端加密云同步，v2.0 可通过手动导出/导入实现。

**Q: 和直接编辑 ~/.ssh/config 有什么区别？**
A: ~/.ssh/config 不支持分组、标签、加密密码存储，且编辑全靠手动。SSH Config Master 提供结构化管理和可视化界面，同时保持与 ~/.ssh/config 的双向同步。
